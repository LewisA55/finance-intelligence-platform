# Phase 8 - CFO Modelling Layer

Project Atlas now treats financial modelling as a governed warehouse-serving
contract rather than an Excel-only convenience layer.

The CFO modelling layer provides model-ready actuals, opening balance sheet
anchors, and readiness classifications that can serve the same consumers:

- Excel integrated three-statement and DCF model
- React model-readiness and valuation pages
- Power BI command-center and modelling packs
- Grounded AI commentary prompts

## Gold Modelling Surfaces

| Model | Purpose |
| --- | --- |
| `mart_model_actuals_feed` | Canonical Company-Total monthly feed for revenue, billings, cash collection, deferred revenue, AR/AP, payroll, ARR/retention, and model-readiness status. |
| `mart_opening_balance_sheet` | Synthetic opening statutory balance sheet anchor for the integrated three-statement model. |
| `mart_model_readiness_controls` | Classification layer that translates noisy source/control observations into modelling treatment: model blocking, accepted limitation, or monitor. |

## Design Principles

- The warehouse supplies actuals, validated histories, and control/readiness
  signals.
- Excel supplies forecast assumptions and integrated statement logic.
- React and Power BI consume the same surfaces for model-readiness and CFO
  storytelling.
- PP&E, capex, debt, tax, and equity forecasting assumptions can remain in the
  Excel model until the platform needs dedicated analytics for those domains.

## Model-Serving Feed Grain

`mart_model_actuals_feed` is one row per:

- `reporting_month_date`
- `reporting_scope`
- `scenario_code`

The initial implementation is scoped to `Company Total` and `ACTUALS`, with
scheduled rows retained where the source layer already contains forward revenue
recognition or workforce plan data.

## Opening Balance Sheet

The opening balance sheet source lives at:

`data/raw/accounting/opening_balance_sheet.csv`

It is generated by `scripts/generators/modeling_layer_generator.py`, loaded
through Bronze, typed in Silver, and exposed as `mart_opening_balance_sheet`.
This avoids using an Excel plug to start the three-statement model.

## Model Readiness Controls

The readiness mapping lives at:

`data/raw/governance/model_readiness_controls.csv`

It is generated by `scripts/generators/modeling_layer_generator.py` so the
policy survives a clean source regeneration even though `data/raw` is ignored
by git.

Examples:

| Control | Treatment |
| --- | --- |
| Revenue recognition governance exceptions | Usable with review |
| Deferred revenue rollforward exceptions | Model Blocking |
| SaaS ARR bridge control issues | Accepted Limitation |
| Workforce HRIS/payroll control issues | Accepted Limitation |
| O2C allocation defects | Monitor |

The aim is to make control noise legible: some issues block modelling, some are
accepted limitations, and some are monitoring signals.

Revenue recognition governance exceptions are deliberately not automatic model
blockers. They should be disclosed and reviewed before sign-off, but the Excel
forecast model can still use the period unless the exception trend indicates a
materially unreliable recognised-revenue baseline.

`mart_model_actuals_feed` also carries:

- `is_actual_period`
- `is_scheduled_period`
- `has_revenue_actuals`
- `has_o2c_actuals`
- `has_financial_actuals`
- `has_saas_actuals`
- `use_in_excel_actuals_flag`

Excel v1.0 should import rows where `use_in_excel_actuals_flag = true` for its
historical actuals/trend basis. The flag requires an actual period, complete
core financial/O2C/revenue actuals, and no model-blocking controls. Scheduled
rows and partial operational histories remain in the feed for React, Power BI,
and forward-looking model-readiness views.
